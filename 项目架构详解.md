# MIA 项目架构详解

> 从零开始理解 MIA（Modular Intelligent Agent）——基于 LLM 决策循环的多 Agent 对话系统。

---

## 一、项目概览

MIA 模拟人类"理解 → 检索 → 规划 → 执行 → 回复"的思考链路。
每启动一次 `python -m mia`，5 个 Agent 通过一条 MessageBus 协作完成对话。

### 办公室类比

| 角色 | Agent | 职责 |
|------|-------|------|
| 🧑‍💼 前台 | ReceiverAgent | 听懂用户的文字/图片/语音（多模态理解） |
| 📋 秘书 | MemoryAgent | 翻档案（检索记忆），做笔记（提取知识） |
| 🧠 经理 | SchedulerAgent | 核心大脑——分析后决定"直接回"还是"先去查" |
| 🔧 执行员 | TaskAgent | 经理说搜新闻就搜新闻，说算数就算数 |
| 📣 发言人 | SenderAgent | 把回复打印到终端（文字或语音播放） |
| 📱 快递员 | WeChatAgent | 微信收发——长轮询收消息，CDN 发回复 |

### 消息流转全景

```
微信/CLI/API → ReceiverAgent → MemoryAgent → SchedulerAgent ⇄ TaskAgent → 按渠道路由
                                   ↑                                    ├── SenderAgent → 终端
                                   └── CONVERSATION_DONE ───────────────┤
                                                                        └── WeChatAgent → 微信
```

Scheduler 根据 `session_id` 前缀自动识别渠道（`wechat:*` → 微信，无前缀 uuid → 终端），回复只走原路返回。

## 二、消息总线（bus/）

### 2.1 设计思想

所有 Agent 通过 MessageBus 通信，不直接调用对方的方法。相当于公司内部群聊——每个人只说自己的事，别人听到跟自己有关的就处理。

### 2.2 消息格式（message.py）

每条消息就是一张快递单：

| 字段 | 含义 | 示例 |
|------|------|------|
| `msg_type` | 包裹类型（19 种枚举值） | `USER_INTENT`, `SEND_TEXT`, `EXECUTE_TASK` |
| `source` | 寄件人 | `"receiver"`, `"scheduler"` |
| `target` | 收件人 | `"memory_agent"`, `"sender"` |
| `payload` | 包裹内容（dict） | `{"intent": "用户想搜新闻", ...}` |
| `session_id` | 会话 ID（12 位 hex） | 关联多轮对话 |
| `parent_id` | 父消息 ID | 追踪任务链 |

工厂函数（`make_user_intent()`, `make_send_text()` 等）提供类型安全的快捷创建方式——source 和 target 被写死，调用方只需填内容。

### 2.3 总线实现（bus.py）

```
MessageBus
├── "receiver"      → asyncio.Queue([msg1, msg2, ...])  ← 每人一个信箱
├── "memory_agent"  → asyncio.Queue([...])
├── "scheduler"     → asyncio.Queue([...])
├── "sender"        → asyncio.Queue([...])
├── "task_agent"    → asyncio.Queue([...])
└── "main"          → asyncio.Queue([...])
```

三个核心操作：

| 操作 | 含义 | 说明 |
|------|------|------|
| `subscribe(name)` | 开信箱 | Agent 启动时调用 |
| `publish(msg)` | 投快递 | 根据 `msg.target` 投到对应信箱 |
| `receive(name, timeout)` | 取快递 | FIFO 顺序，超时返回 None |

定向消息只投给 `target` 指定的订阅者；广播消息（`target="broadcast"`）投给所有人（除了发送方自己）。队列满时丢弃最旧的消息。

## 三、Agent 基类（agents/base.py）

所有 Agent 的"员工手册"，定义了统一的生老病死：

```
__init__(name, bus)          ← 出生：拿到工牌和群聊地址
      │
      ▼
start()                      ← 上班：subscribe → on_start() → 广播 SYSTEM_READY
      │
      ▼
run()                        ← 工作中（后台循环）：
  └── while _running:        ←   不断循环
        ├── bus.receive()    ←     看信箱（1 秒超时）
        └── handle(msg)      ←     处理消息（子类实现）
      │
      ▼
stop()                       ← 下班：广播 SHUTDOWN → on_stop() → unsubscribe
```

关键设计：
- `handle()` 是抽象方法（`@abstractmethod`）——每个 Agent 必须自己实现，这是唯一不同的地方
- `send()` 是快捷方式——自动填 `msg.source = self.name` 然后 publish
- `run()` 的 1 秒超时是为了能及时响应 `_running = False`（否则会永远卡在 receive）
- `on_start()` / `on_stop()` 是可选钩子——MemoryAgent 用 on_start 加载持久知识

## 四、ReceiverAgent — 前台（agents/receiver.py）

**职责**：把用户的乱七八糟输入（文字/图片/语音）翻译成标准化的 `USER_INTENT`。

### handle() 处理流程

```
收到 RAW_INPUT
      │
      ├── 有图片？──→ MiMo VL 理解图片内容 → 拼入意图
      │
      ├── 有语音？──→ MiMo 多模态音频理解（内容+情绪+意图）→ 拼入意图
      │         └── 纯语音（无文字）？→ 告诉 Scheduler"把这当用户原话"
      │
      └── 无语音？──→ 纯文字，直接拼接
      │
      ▼
   创建 USER_INTENT → send() → 飞向 MemoryAgent
```

### 多模态能力

| 输入类型 | 使用方法 | 模型 | 能力 |
|---------|---------|------|------|
| 图片 | `mimo.understand_image()` | mimo-v2.5 | 视觉理解（描述图片内容） |
| 音频 | `mimo.understand_audio()` | mimo-v2.5 | 多模态理解（转写+情绪+意图） |
| 音频（降级） | `mimo.transcribe()` | mimo-v2.5-asr | 纯 ASR 转写 |

Receiver 依赖 MiMoProvider 处理图片和音频——读文件 → 转 base64 → 调 API → 返回理解结果。

## 五、MemoryAgent — 秘书（agents/memory.py）

最复杂的 Agent，有"双面人生"：

```
收到 USER_INTENT          → 面① "翻书"：检索记忆 → 注入上下文 → 转发给 Scheduler
收到 CONVERSATION_DONE    → 面② "记笔记"：提取知识 → 存临时记忆 → 整理入库
```

### 5.1 面①：检索与注入

`_on_user_intent()` 构造 `memory_context`，包含两部分：

**Part 1 — 对话历史**：最近 N 轮（默认 5）user+assistant 原文，帮助 LLM 理解指代和上下文：

```
## 对话历史
用户: 帮我推荐几本 Python 入门书
助手: 推荐《Python编程：从入门到实践》...
用户: 哪本适合完全零基础的？
```

**Part 2 — 知识记忆**：从两级记忆中检索到的知识点：

```
## 相关记忆
- [偏好] 用户是编程初学者，偏好中文资料（置信度: 0.8）
- [事实] 用户正在学习 Python，已掌握基础语法（置信度: 0.7）
```

两部分拼成 `memory_context`，注入到转发消息的 payload 中。Scheduler 不需要知道记忆系统如何工作——它只需读 `payload["memory_context"]`。

### 5.2 面②：两级记忆系统

```
每次对话结束
      │
      ▼
  Level 1 提取（内存，快速，轻量）
  ├── LLM 提取 1-3 条原子知识
  ├── confidence = 0.5（临时，不确定）
  └── 存到 _working_memory 列表
      │
      │  触发条件：积累 ≥ 30 条  或  检测到换日
      ▼
  Level 2 合并（磁盘，重量级，高精度）
  ├── LLM 合并去重所有临时记忆 + 原始对话
  ├── confidence ≥ 0.7（持久，较确定）
  └── 存到 MemoryStore 磁盘文件
```

类比：Level 1 是桌面便签纸（随手记，可能重复），Level 2 是周末整理的笔记本（去重、分类、装订）。

### 5.3 知识提取（Level 1）

`_extract_working_knowledge()` 用 LLM 从单轮对话中提取 1-3 条原子知识。知识类别：

| 类别 | 含义 | 示例 |
|------|------|------|
| `fact` | 事实 | "用户的家乡在嘉兴" |
| `preference` | 偏好 | "用户偏好中文技术文档" |
| `decision` | 决策 | "用户决定明天开始学 Rust" |
| `task` | 任务 | "用户需要完成一个 Web 项目" |
| `insight` | 洞察 | "用户对性能优化特别感兴趣" |

若 LLM 超时或失败，降级到 `_local_extract_knowledge()`——纯本地规则提取，不依赖 API，保证知识不丢失。降级条目置信度 0.3，标记为待 LLM 验证。

### 5.4 知识合并（Level 2）

`_consolidate_daily()` 将 `_working_memory` + `_daily_buffer` 提交给 LLM：

1. **合并**：相同主题的多条合并为一条
2. **去重**：重复只保留最新版
3. **更新**：新信息覆盖旧信息（用户偏好可能变）
4. **废弃**：问候、道别等低价值信息丢弃
5. **置信度提升**：多次出现的知识置信度更高（≥ 0.7）

若 LLM 超时或失败，降级到 `_fallback_persist()`——直接将临时记忆持久化（不合并），confidence 至少 0.6。

### 5.5 降级策略总结

| 环节 | 正常路径 | 降级路径 | 保证 |
|------|---------|---------|------|
| Level 1 提取 | LLM 提取 | 本地规则提取 | 知识不丢失 |
| Level 2 合并 | LLM 合并去重 | 直接持久化 | 数据落盘 |
| 检索 | keyword + LLM rerank | 纯 keyword 匹配 | 检索有结果 |
| /compact | LLM 摘要 | 返回降级消息 | 不抛异常 |

### 5.6 持久存储（memory/store.py）

两级文件架构，北京时间（UTC+8）：

```
data/memory/
├── index.json          ← 始终加载（~1-2 KB），每天一条 DaySummary
└── daily/
    ├── 2026-06-18.json ← 按需懒加载，当日 KnowledgeEntry 列表
    └── 2026-06-19.json
```

检索流程：`scan_index(keywords)` → 缩小到相关日期 → `load_day(date)` → keyword match + LLM rerank。

## 六、SchedulerAgent — 经理（agents/scheduler.py）

整个系统的核心大脑。690 行，核心是一个 LLM 决策循环。

### 6.1 决策循环

```
收到 USER_INTENT / TASK_RESULT
    ↓
┌─────────────────────────────┐
│  _run_loop():               │
│                             │
│  ① 安全检查（≤10 轮，≤3 连续任务）│
│  ② _build_context(): 拼上下文  │
│  ③ provider.chat_sync(): 问 LLM│
│  ④ _parse_decision(): 解析 JSON│
│  ⑤ _execute_decision():       │
│     ├── "reply" → 流式/语音回复 │
│     ├── "execute_task" → 派任务 │→ 等结果 → handle() → _run_loop()
│     └── "done" → 结束           │
└─────────────────────────────┘
```

### 6.2 Prompt 外部化

系统 prompt 从文件加载（`prompts/` + `AGENTS.md`），支持自定义：

| 文件 | 用途 | 何时用 |
|------|------|--------|
| `AGENTS.md` | MIA 身份定义（名字、性格、口吻） | 所有 LLM 调用前注入 |
| `prompts/scheduler.md` | 决策指令（JSON 格式、规则） | `_build_context()` |
| `prompts/reply.md` | 回复指令（语气、长度、格式） | `_build_reply_context()` |

决策时是"分析者"（scheduler.md），回复时是"对话者"（reply.md）——两个角色，两套 prompt，两个独立的 LLM 调用。

### 6.3 决策 JSON 格式

LLM 返回三种 action：

```json
// 直接回复
{"action": "reply", "action_detail": {"use_voice": false}}

// 派发任务
{"action": "execute_task", "action_detail": {"task": "搜索科技新闻", "tools_hint": ["web_search"]}}

// 结束
{"action": "done", "action_detail": {}}
```

`_parse_decision()` 用正则从 LLM 输出中提取 JSON——先找 ` ```json ``` ` 代码块，再找 `{...}` 边界，容错性强。解析失败会重试一次（带上格式纠正指令）。

### 6.4 流式回复

`_stream_reply()` 实现逐字输出：

```
STREAM_START → Sender 打印前缀（不换行）
STREAM_CHUNK ×N → Sender 逐 token 打印（flush 立即显示）
STREAM_END → Sender 换行 + 发 CONVERSATION_DONE
```

包含完整 fallback 链：主 Provider 流式 → 备选 Provider 流式 → 错误文本降级。流式关闭时（`enable_streaming=false`）降级为非流式 `_generate_fallback_reply()`。

### 6.5 渠道感知路由

Scheduler 不再广播回复到所有输出目标。`session_id` 编码了消息来源渠道：

| session_id 格式 | 来源 | 回复目标 |
|-----------------|------|----------|
| `wechat:<user_id>` | 微信 | `target="wechat"` (WeChatAgent) |
| `<uuid_hex>` (无前缀) | CLI/API | `target="sender"` (SenderAgent) |

`_resolve_output_target()` 方法根据前缀自动判断。未来加 Telegram/Discord 只需加一行映射。

### 6.6 安全保护

| 限制 | 值 | 触发后的行为 |
|------|-----|------------|
| 最大迭代轮数 | 10 | 强制回复，不继续思考 |
| 连续任务上限 | 3 | 强制回复，停止派发新任务 |
| 单任务超时 | 60s | 继续循环，LLM 决定下一步 |
| 重复任务检测 | `_task_history` | 跳过，生成虚拟结果继续 |

## 七、TaskAgent — 执行员（agents/task.py）

Scheduler 的"兄弟"——结构相同的 LLM 循环，但更短更简单（331 行）。

### 7.1 工具调用循环

```
收到 EXECUTE_TASK
      │
      ▼
┌────────────────────────────┐
│  for iteration in 1..5:    │
│                            │
│  ① _call_llm(): 问 LLM     │
│  ② _parse_decision(): 解析  │
│                            │
│  if action == "finish":    │
│      return 结果           │
│                            │
│  elif action == "call_tool":│
│      tool.execute(...)     │
│      结果追加到 messages    │
│      继续循环              │
└────────────────────────────┘
      │
      ▼
  TASK_RESULT → Scheduler
```

关键设计：**messages 列表不断增长**——每次工具调用的结果都追加进去，LLM 能看到"已经做了什么"。

### 7.2 内置工具

| 工具 | 实现 | 功能 |
|------|------|------|
| `web_search` | DuckDuckGo | 搜索互联网信息 |
| `weather` | 在线 API | 查询城市天气 |
| `shell` | subprocess | 执行 Shell 命令 |
| `file` | 本地文件系统 | 读写文件 |

每个 Tool 是 `Tool` ABC 的子类，定义 `name`、`description`、`parameters`（JSON Schema）和 `execute()`。工具描述被拼成 Markdown 发给 LLM，让它知道何时用什么工具。

### 7.3 安全限制

| 限制 | 值 | 说明 |
|------|-----|------|
| 最大工具调用 | 5 次 | 用完强制返回已有结果 |
| 同类工具上限 | 2 次 | 防止反复搜索同一话题 |
| 工具不存在 | — | 反馈 LLM "工具不存在"，让其换一个 |

### Scheduler vs TaskAgent 对比

| | Scheduler | TaskAgent |
|------|-----------|-----------|
| 角色 | 战略（做什么） | 战术（怎么做） |
| 输入 | USER_INTENT / TASK_RESULT | EXECUTE_TASK |
| 最大轮数 | 10 | 5 |
| 决策选项 | reply / execute_task / done | call_tool / finish |
| 流式输出 | ✅ | ❌ |
| 记忆上下文 | ✅ MemoryAgent 注入 | ❌ |

本质相同的 Agent Loop 模式：**收到消息 → LLM 循环 → 解析 JSON → 执行决策 → 循环或返回**。

## 八、SenderAgent — 发言人（agents/sender.py）

最末端的 Agent，把回复输出给用户。

### 消息分拣

```
handle() 收到消息
      │
      ├── SEND_TEXT      → 打印完整文字 → 发 CONVERSATION_DONE
      │
      ├── SEND_VOICE     → TTS 合成 → 保存文件 → 后台自动播放 → 发 CONVERSATION_DONE
      │
      ├── STREAM_START   → 打印前缀（不换行）
      ├── STREAM_CHUNK   → 打印增量文字（不换行，flush 立即显示）
      └── STREAM_END     → 换行 → 发 CONVERSATION_DONE
```

### 语音回复流程

1. 调用 `mimo.synthesize(text, voice, format)` 生成音频 bytes
2. 保存到 `workspace/reply_{msg_id}.wav`
3. 通过 `loop.run_in_executor()` 在后台线程播放（`audio/playback.py`）
4. 播放失败不影响主流程（静默失败）

### CONVERSATION_DONE 双发

Sender 发送两份 `CONVERSATION_DONE`：
- `target="main"` → 主程序用来到达等待循环，继续下一轮对话
- `target="memory_agent"` → 秘书用来提取记忆

## 八-B、WeChatAgent — 微信渠道（channels/wechat/agent.py）

MIA 和微信之间的桥梁 Agent。通过腾讯 iLink Bot API 接入微信个人号。

### 架构位置

```
微信用户 ←→ (iLink API) ←→ WeChatAgent ←→ MessageBus ←→ MIA Agent 链路
```

WeChatAgent 和 SenderAgent 是**平级的两个"出口"**——Scheduler 决定回复后，根据 session_id 前缀只发给其中一个。

### 入站：微信 → MIA

```
微信 App 发消息
  → iLink 长轮询 (后台线程, 最长 hold 35s)
  → _on_message() 解析消息:
      ├─ type=1 文本 → 直接作为用户输入
      ├─ type=2 图片 → CDN 下载 (AES-128-ECB 解密) → 本地文件
      ├─ type=3 语音 → ASR 文本 + CDN 下载原始音频 → pilk SILK→WAV 转码
      ├─ type=4 文件 → 文件信息提示
      └─ type=5 视频 → 视频提示
  → 去重 (ID 级 + 内容级双重去重)
  → publish RAW_INPUT (session_id="wechat:<user_id>")
  → 进入 MIA 标准 Agent 链路
```

### 出站：MIA → 微信

```
Scheduler publish SEND_VOICE(target="wechat")
  → _handle_output_voice():
      1. MiMo TTS 合成 (mimo.synthesize)
      2. CDN 上传 (upload_media, AES-128-ECB 加密)
      3. file_item 发送 (用户收到可点击播放的音频文件)
      4. 文本 fallback (🎤 前缀)
```

### 认证

- 首次运行无 token → 终端打印 QR 码 URL → 微信扫码 → token 持久化到 `~/.mia/wechat_bot_token`
- 后续启动自动加载 token

### 依赖

- `httpx` — HTTP 客户端
- `pycryptodome` — AES-128-ECB 媒体加解密
- `pilk` — SILK V3 语音解码

### 文件结构

```
channels/wechat/
├── agent.py   # WeChatAgent (长轮询 + 消息收发 + SILK 转码)
├── client.py  # ILinkClient (iLink HTTP API 封装)
└── utils.py   # AES 加解密 + HTTP 请求头
```

## 九、配置系统（config.py）

`pydantic-settings` 从 `.env` 文件和环境变量加载，三个配置组：

| 配置组 | 环境变量前缀 | 管什么 |
|--------|------------|--------|
| `MiMoConfig` | `MIMO_` | MiMo API Key、Base URL、各模型名 |
| `DeepSeekConfig` | `DEEPSEEK_` | 备选 Provider |
| `AgentConfig` | `MIA_` | Scheduler 安全限制、记忆参数、流式开关 |

通过 `get_config()` 全局单例访问。MiMoConfig 自动识别 key 类型：`tp-` 开头走 Token Plan 网关，`sk-` 开头走按量付费网关。

## 十、音频子系统（audio/）

可选模块（`pip install -e ".[audio]"`），依赖 `sounddevice` + `soundfile` + `numpy`：

| 模块 | 函数 | 功能 |
|------|------|------|
| `audio/recorder.py` | `record_until_keypress()` | 麦克风录音，按 Enter 开始/停止，16kHz mono WAV |
| `audio/playback.py` | `play_audio(filepath, blocking)` | 跨平台音频播放，支持 WAV/FLAC/OGG |

录音文件存于 `%TEMP%/mia_recordings/`，播放默认非阻塞（TTS 自动播放用后台线程）。

## 十一、Provider 架构（providers/）

`BaseProvider` 定义 `chat()`, `chat_sync()`, `chat_stream()` 三个接口。MiMoProvider 和 DeepSeekProvider 均基于 `openai.AsyncOpenAI`。

**MiMoProvider 完整能力**：

| 方法 | 模型 | 功能 |
|------|------|------|
| `chat()` / `chat_sync()` / `chat_stream()` | mimo-v2.5-pro | 文本对话（OpenAI 兼容） |
| `understand_image()` | mimo-v2.5 | VL 图片理解 |
| `understand_audio()` | mimo-v2.5 | 多模态音频理解（转写+情绪+意图） |
| `transcribe()` | mimo-v2.5-asr | 纯 ASR 语音转文字 |
| `synthesize()` | mimo-v2.5-tts | TTS 文字→语音（非流式） |
| `synthesize_stream()` | mimo-v2.5-tts | TTS 流式合成 |

所有 LLM 调用位置都有 **Provider fallback** 链：主 Provider → 备选 Provider → 错误降级。

## 十二、核心设计模式

### 12.1 Agent Loop 模式

```
while 还没完 and 还有次数:
    问 LLM（带上历史）
    解析 JSON 决策
    执行决策
    if 该结束了:
        返回
    else:
        把结果追加到对话历史
        继续
```

SchedulerAgent 和 TaskAgent 都是这个模式的实例——区别只在于 LLM 的角色（战略 vs 战术）和决策选项（3 种 vs 2 种）。

### 12.2 中间件模式

MemoryAgent 不改变消息类型，只增强 payload——截获 `USER_INTENT` → 注入 `memory_context` → 转发。Scheduler 完全不知道记忆系统的存在。

### 12.3 降级策略

系统的韧性设计：每个依赖外部 API 的环节都有降级路径——知识提取失败用本地规则，合并失败直接持久化，检索失败取最近 N 条，流式失败用非流式，主 Provider 失败用备选。

### 12.4 流式推送模式

`STREAM_START → STREAM_CHUNK×N → STREAM_END` 三段式协议，让 Sender 能逐字打印而不需要知道 LLM 的内部实现。完整文本在 `STREAM_END` 时附带，供 MemoryAgent 存储。

### 12.5 工厂函数模式

`message.py` 中的 `make_xxx()` 函数封装了消息创建逻辑——写死 source/target，暴露业务参数，保证类型安全。

## 十三、文件导航

```
src/mia/
├── main.py              ← 入口：CLI 交互 + HTTP API 服务
├── config.py            ← 配置管理（pydantic-settings）
├── bus/
│   ├── message.py       ← 消息类型定义 + 工厂函数
│   └── bus.py           ← 异步消息总线（asyncio.Queue）
├── agents/
│   ├── base.py          ← Agent 基类（生命周期模板）
│   ├── receiver.py      ← 多模态输入理解
│   ├── memory.py        ← 两级知识记忆 + 对话历史
│   ├── scheduler.py     ← LLM 决策循环 + 渠道感知路由
│   ├── task.py          ← LLM 工具调用循环
│   └── sender.py        ← 终端输出（文字/语音/流式）
├── channels/            ← 通信渠道（可选，按需加载）
│   └── wechat/
│       ├── agent.py     ← WeChatAgent（iLink 长轮询 + 收发）
│       ├── client.py    ← ILinkClient（iLink HTTP API 封装）
│       └── utils.py     ← AES-128-ECB 加解密 + 请求头
├── memory/
│   ├── store.py         ← 知识存储（index + daily JSON）
│   ├── retriever.py     ← 关键词 + LLM 混合检索
│   └── browser.py       ← TUI 记忆浏览器
├── providers/
│   ├── base.py          ← Provider 抽象接口
│   ├── mimo.py          ← MiMo API 封装（Chat+VL+ASR+TTS）
│   └── deepseek.py      ← DeepSeek API 封装（备选）

│   ├── base.py          ← Tool 抽象类

│   ├── weather.py       ← 天气查询
│   ├── shell.py         ← Shell 命令执行
│   └── file.py          ← 文件读写
└── audio/
    ├── recorder.py      ← 麦克风录音
    └── playback.py      ← 音频播放

prompts/
├── scheduler.md         ← Scheduler 决策 prompt

└── task_agent.md       ← TaskAgent 执行 prompt

AGENTS.md               ← MIA 身份定义（注入所有 system prompt）
```
